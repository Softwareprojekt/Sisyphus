 Nicht-funktionale Anforderungen
 Benutzbarkeit (Usability)
  • sowohl  mit Maus als auch über Tastatur voll benutzbar -> Sollte ja leicht  zu realisieren sein durch bspw. Shortcuts (zumindest fuer die  grundlegenden Funktionen wie Vor/Zurueck, Hauptmenu, Speichern/Laden...
  • für  die Zielgruppe (Technik-Studenten) einfache Benutzbarkeit -> Hier  braucht man glaub ich nicht naeher drauf eingehen... wenn wirs richtig  machen dann sollten die das auch hinbekommen ;)
  • wenn  irgendwas nicht einfach benutzbar, sollten im Programm kurze  verständliche Anweisungen oder Anleitungen bei der Bedienung helfen  -> gabs da nicht schon ein Tutorial oder sowas?
  • wenn  Rechenvorgänge längerer Zeit (z.B. >3 s) auftreten können, soll der  Fortschritt mit einem Fortschrittsbalken oder Ähnliches verdeutlicht  werden -> Kann sowas vorkommen? Wenn ja, muss es auch abbrechbar  sein... Muss dann noch geklaert werden.
    • einfachere Alternative statt Fortschrittsbalken: Rechenvorgänge mit einem animierten Symbol (z.B. Spinner) verdeutlichen
  • wenn  das System einen längeren Rechenvorgang durchführt, sollte der Benutzer  die Möglichkeit nicht-rechenbezogene Sachen im Arbeitsheft zu ändern und Funktionen außerhalb des Arbeitsheft nutzen können; z.B. Notizen  hinzufügen, Hilfe aufrufen -> Das find ich auf jeden Fall wichtig!  Muessen wir aber dann noch mit der Architektur klaeren!
  • Darstellung von Vektoren und (2-dimensionalen) Matrizen innerhalb von Arbeitsblätter wie in Mathematik üblich -> GUI
    • evtl. nur Nice-To-Have
  • in  Arbeitsblättern Erkennbarkeit für den Benutzer welche Eingaben erfolgen  sollten -> Das sollte auch noch in das Tutorial rein? Sonst auch in  die GUI
    • (kann in den dafür vorgesehenen Use-Case)
  • Rückmeldung  wenn Berechnungen nicht durchgeführt werden können, da mathematisch  nicht berechenbar (Bsp. Determinante einer Matrix gleich 0)
    • (kann in den dafür vorgesehenen Use-Case)
  • Popup's von Dialogen gering halten -> PopUps sind sowieso verboten! ;) Wird auch auf die GUI weiter geschoben.
  • farbliche  Hervorhebungen oder farbige Schriften von Vektoren - die Ergebnisse  oder Eingabefelder kennzeichen - sollten änderbar sein

 Zuverlässigkeit (Reliability)
  • Das System sollte sich ohne Probleme so beenden lassen, dass keine programmbezogenen Prozesse geöffnet bleiben →im Use-Case Beenden wohl schon erfasst (?) -> Da sollte aber dann auch richtig aufgeraeumt werden...
  • System  frei von kritischen Fehlern (keine Speicherzugriffsfehler, Abstürze...)  -> Das erledigt sich hoffentlich durch C# selbst
  • es  sollen vom Benutzer keine Eingaben durchgeführt werden können, durch  die das System dann längere Zeit nicht mehr reagiert -> Hier wieder  das Abbrechen von Berechnungen das Stichwort.
    • vgl. hier mit Leistung (Performance) Anforderungen
    • relevant  für Fraktalberechnungen die einige Iterationen benutzen -> Muessen  Fraktale berechnet werden oder verwenden wir vorberechnete Bilder?


 Leistung (Performance)
  • Rechenvorgänge der Anwendung sollten i.d.R. höchstens 3 Sekunden betragen -> s.o. (Abbrechbar)
    • evtl. Ausnahmen bei Fraktalberechnung möglich (?)


 Unterstützbarkeit (Supportability)
  • über  Plugins (DLL-Bibliotheken) sollte es möglich sein die möglichen  Aufgabentypen zu erweitern bzw. zu ändern -> Hier muss auf jeden Fall  die Schnittstelle so frueh wie moeglich geklaert werden. Auch die  Interfaces muessen dafuer stehen.
  • kompilierbares Programm mit Visual Studio 2012 unter Windows 7, 8
  • kompilierbar mit Mono unter Mac, Linux
    • Nice-To-Have
  • die  Mind-Map sollte anhand einer Strukturdatei (z.B. XML) aufgebaut werden  -> Wenn die Aufgaben ueber DLLs eingebunden werden dann koennen wir  auch daraus selber eine XML Datei generieren. Sollte man auch klaeren...
